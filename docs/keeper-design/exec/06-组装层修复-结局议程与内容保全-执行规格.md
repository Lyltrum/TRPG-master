# 执行规格 · 组装层修复：结局/议程被吞 + 静默内容丢失（4a 收尾）

> 承接 [05-预处理管线组装与校验](05-预处理管线组装与校验-执行规格.md)。
> 4a 端到端跑通了，但组装质量有两个真缺陷，会污染 4b 的缺口分析——必须先修，
> 拿到一份**组装质量合格**的科比特产物，4b 才有干净地基。分支 `feat/keeper-agent`。

## 0. 两个缺陷（已定位根因，不是猜）

### 缺陷 A：endings=0 / agenda=0，材料被吞进 kp_truth
- `ending-adventure-options`（裸抽取 what_kind = **"结局选项说明"**）→ 被归到
  `kp_truth`，理由"科比特真相的一部分"；
- `current-situation`（what_kind = "当前事件描述"，是"今晚行动"的时间压力）→
  同样进 `kp_truth`；
- 根因：**阶段1 把 `kp_truth` 当兜底垃圾桶**——该建 ending/agenda 实体时没建，
  阶段2 于是产出 0 个。schema 有 endings/agenda 位，是组装层没用。

### 缺陷 B：9 个调查线索片段塞进 player_intro → 内容静默丢失
- 9 个**调查线索**片段（报纸剪报、证词、跟踪类行动等）被归到 `player_intro`；
- 这些是玩家要**挣得**的内容，该进 node（在某个调查节点里被发现）；
- `player_intro` 是**唯一直接念给玩家的单个字符串**（本产出仅 274 字）——9 个
  片段的内容根本装不下，**在阶段2/3 蒸发了**；
- 根因：**校验的"无孤儿"只验了阶段1 的分配、没验阶段2/3 的内容保全**。片段
  "被归组"了，但归到一个装不下的薄字段里，内容悄悄消失。**这是校验漏洞。**

## 1. 修复一：阶段1 归组的三条硬约束（改 prompt + 校验兜底）

阶段1 的 prompt 明确以下（它已经能看到 schema，这是 B）：

1. **`kp_truth` 不是兜底桶**：只装"背景真相/超自然设定"这类**纯背景**片段。
   一个片段如果本身就是"结局""事件""线索""地点""NPC"，归到它的结构位，
   **不许因为"和真相有关"就丢进 kp_truth**（几乎所有片段都和真相有关）。
2. **薄公开字段各只收 1 个片段**：`player_intro` / `opening` / `meta` 各自
   **最多归入 1 个**片段（那个真正"公开开场框架"的片段）。**调查线索
   （报纸/证词/现场发现）是玩家挣得的内容，一律进 node，不进 player_intro。**
3. **结局与时间压力要建实体**：what_kind 指示"结局/结尾"的片段 → 建 `ending`；
   指示"当前事件/行动规律/时间/今晚/期限"这类时间驱动的 → 建 `agenda` 条目
   （trigger 用自由文本，见 04 规格）。

阶段1 应利用裸抽取里每个片段的 **`audience` 字段**辅助归组：`守密人绝密` 的
片段不该进任何玩家可见字段。

## 2. 修复二：校验闭环补三项（补 `validate_module.py`，接进自修循环）

现有五项保留。新增：

| 新校验 | 判据 | 抓的缺陷 |
|---|---|---|
| **薄公开槽上限** | 归到 `player_intro`+`opening`+`meta` 的片段合计 **≤ 3**（各 ≤1） | 缺陷 B（线索倒灌开场白） |
| **绝密不进公开槽** | `audience` 含"绝密/守密人/KP"的片段，不得归到 `player_intro`/`opening` | 静默丢失 + 泄密双重风险 |
| **结构完整性** | 若存在 what_kind 含"结局/结尾/结束"的片段却 `endings==[]`，硬失败；含"当前事件/行动规律/时间压力/今晚/期限"却 `agenda==[]`，硬失败 | 缺陷 A（结局/议程被吞） |

🔴 **结构完整性校验是"完整性检查"不是"路由逻辑"**：路由仍由 LLM（阶段1）做，
代码只机械核对"带结局信号的片段没有全被丢掉"。用片段自己的 `what_kind_of_thing`
当信号，不预设 COC 专属路由。

这三项接进 05 已有的自修循环（校验失败→错误喂回阶段1 重新归组→再校验，≤2 次）。

## 3. 内容保全的根本补强（比上面三项更重要）

"无孤儿"验的是**分配**，不是**保全**。补一项**内容保全校验**：

- 对每个源片段，检查它的**内容是否在最终模组里有落点**——机械代理判据：
  片段的归宿字段在最终 JSON 里**非空**，且若归宿是"多片段合一"的薄字段
  （player_intro/opening/meta），该字段**不得**是多个实质片段的归宿（见修复二）；
- 更强的代理：统计**每个片段的 summary 关键词**有多少在最终模组对应实体的
  文本里能找到（宽松子串匹配，阈值可调，目的是抓"整段蒸发"不是逐字比对）；
  低于阈值的片段列进"疑似内容丢失"清单。
- 这项**不必做成硬失败**（关键词匹配有噪声），但要**写进校验报告**让人能看到
  哪些片段的内容可能没落地。
