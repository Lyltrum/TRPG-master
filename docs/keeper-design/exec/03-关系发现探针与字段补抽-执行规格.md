## A · chunk 2 字段补抽

**只补字段，不重抽条目。** 重抽会产生不同的条目边界，破坏已有的行号与
覆盖率，也让两批数据不可比。

- 读 `模组资料/科比特先生.裸抽取.json`，取出 `chunk_index == 2` 且
  `what_kind_of_thing` 为空的条目；
- 把每条的 `title` + `summary` + 它对应的**原文行区间正文**喂回 DeepSeek，
  只问一个问题：**「用你自己的话说，这是一段什么东西」**；
- 🔴 同样**不给任何候选词、不给上一轮已有的描述清单**——给了就是提示，
  它会往已有的类上靠，补出来的字段就失去了独立性；
- 结果写回原 JSON 的对应条目（原地更新，其余字段一字不改），
  并在条目上标 `"what_kind_of_thing_source": "backfill"` 以便区分。

## B · 关系发现探针

### 核心难题：怎么问「关系」而不暗示 `leads_to`

问"哪些条目**通向**哪些条目" → 预设了有向链。
问"条目之间的**邻接**" → 预设了图。
两个都是在拿现有 schema 的形状去套。

**正确问法**：给出全部条目的清单，让模型指出**哪两条之间存在联系**，
并**用它自己的话描述这个联系是什么**——`relation` 是自由文本，
与第一轮的 `what_kind_of_thing` 同构。关系的**类型**必须自己浮现。

### 具体做法

一次调用给全局视野（59 条的 `id` + `title` + `what_kind_of_thing` +
`summary` 大约几千 token，装得下；关系发现需要看到全貌，分块问会丢跨块关系）。

输出每条关系：

```jsonc
{
  "item_a": "条目 id",
  "item_b": "条目 id",
  "relation": "用你自己的话描述这两者之间是什么联系"
}
```

🔴 **字段设计的三条纪律**：

1. **不要 `from`/`to`**，用中性的 `item_a`/`item_b`——`from→to` 本身就在
   暗示有向。方向性如果存在，让它在 `relation` 的描述里自然说出来
   （"甲是乙的前置" / "甲乙是同一处地方的两部分"，方向藏在措辞里）。
2. **不要问 `directional: true/false`**，不要问关系强度、不要给类型枚举。
3. **允许"没有关系"**：prompt 里明确说，某个条目跟其它任何条目都没有联系
   是完全正常的答案。孤立条目本身是信息——它说明那类东西是独立实体。

### 🔴 这一轮要做两遍独立跑 + 对拍

与 02 规格「不做对拍」不冲突，理由是这一轮的性质不同：

- 关系比实体**主观得多**，臆造风险高；
- 成本极低（每遍 1 次调用）；
- 两遍都出现的关系 = 可信；**只出现一遍的 = 存疑，那恰恰是我要看的边界**。

两遍用**不同的随机种子/温度**（建议 temp 0.3 与 0.5）跑同一个 prompt，
各自独立存盘，再机械对拍：

- 交集（两遍都有的 `{item_a, item_b}` 无序对）
- 差集（只有一遍有的）
- 同一对但 `relation` 描述不同的（**这类最有价值**：说明关系存在但性质有歧义）

对拍是**机械比较，不做语义合并**——"甲是乙的前置"和"乙依赖甲"算不算同一个
关系，那是我要判断的，脚本不许替我归并。

## 1. 汇报要求（我验收时看的就是这些）

**每条命令的输出真实可见——不要重定向到 `/dev/null`，不要串长 `&&` 链。**

   归类是我的活，你归了就污染实验；
