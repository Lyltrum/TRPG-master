## 0. 目标与边界

**一句话**：让模组拥有"时间骨架"——开场脚本 + 议程时间轴 —— 并让裁决器
能引用它、触发它、且触发记账由**代码**保证（不靠 prompt 自觉）。

**本期做**
- 模组 schema 新增 `opening` / `agenda` / `endings[].trigger`（向后兼容）；
- 加载器渲染这些块，进裁决/叙事两个 system prompt；
- 每轮把「议程状态（未发生/已发生）」注入局面块；
- 裁决契约新增 `agenda_fired`，执行阶段写进 `keeper_state`，`once` 语义由代码去重。

**本期不做**（属提案②③，不要顺手做）
- 世界心跳 ticker、主动轮、`KEEPER_HEARTBEAT_ENABLED`；
- 对局阶段状态机、`ending_reached`、结局收束流程；
- `trigger.type == "silence"` 的**判定逻辑**（本期只存字段、只渲染，心跳才消费）；
- `endings[].trigger` 的判定（本期只存、只渲染）。

**绝对不要碰**：`trpg-sdk/`、`trpg-frontend/`、`app/dto/`、WS 协议、
alembic 迁移（`rooms.keeper_state` 列已存在，本期不加列）。

## 1. `app/core/keeper/module_loader.py`

### 1.1 新增模型（放在 `ModuleEnding` 之前）

```python
class AgendaTrigger(_LenientModel):
    """议程事件的触发条件。

    type 刻意用 str 而非 Literal：不同模组会长出新的触发轴，未知类型应当
    "渲染成原样让 KP 自行裁量"，而不是加载时炸掉（加载器的职责是读得出来）。
    """

    type: str  # "game_night" | "silence" | "manual"
    at: int | None = None       # game_night：第 N 个游戏内夜晚
    seconds: int | None = None  # silence：现实沉默秒数（提案②世界心跳消费，本期只存）
    note: str | None = None     # manual：时机描述，由裁决器裁量


class AgendaEvent(_LenientModel):
    """时间轴上的一个"该发生的事"——不依赖玩家行动，世界自己会动。"""

    id: str
    title: str | None = None
    trigger: AgendaTrigger
    kp_text: str
    effects: list[str] = []
    once: bool = True


class ModuleOpening(_LenientModel):
    """开场脚本（提案③的 opening 阶段会消费，本期先让它存在并进 prompt）。"""

    scene: str | None = None
    script: str
    kp_notes: str | None = None
```

### 1.2 既有模型扩展

- `ModuleEnding` 加 `trigger: str | None = None`
  （可判定的触发描述；`condition` 保留不动，两者共存——`condition` 是原文
  叙述，`trigger` 是为提案③写的判定口径）。
- `ScenarioModule` 加：
  ```python
  opening: ModuleOpening | None = None
  agenda: list[AgendaEvent] = []
  ```
  并加查询方法（与 `node_by_id` / `npc_by_id` 同风格）：
  ```python
  def agenda_by_id(self, event_id: str) -> AgendaEvent | None: ...
  ```

🔴 **向后兼容是硬要求**：三个新字段全部可选/默认空，现有不含它们的
JSON（fixture、任何旧剧本）必须照常 `load_module` 成功。测试要守这条。

### 1.3 渲染函数（放在 `render_endings` 附近）

```python
def render_agenda_trigger(trigger: AgendaTrigger) -> str:
    """把触发条件渲染成人话。未知 type 原样带出，不抛异常。"""
    # game_night → "第 2 个游戏内夜晚"
    # silence    → "现实沉默 90 秒后"
    # manual     → "KP 裁量：{note}"
    # 其它       → f"{type}（{已填字段的键值对}）"


def render_agenda(module: ScenarioModule) -> str:
    """议程时间轴。每条：id · 标题（触发条件）→ kp_text，effects 缩进列出。"""


def render_opening(module: ScenarioModule) -> str:
    """开场脚本：场景 + 念白 + KP 要建立什么。"""
```

`render_full` 的块顺序改为：

```
overview → 【开场脚本】→ 【议程时间轴（绝密）】→ 调查节点 → NPC → 结局
```

🔴 **空块必须整块省略**：`opening is None` / `agenda == []` 时，
`render_full` 里不能出现空标题（旧模组的输出不应变脏）。

🔴 议程块的标题必须带**绝密**字样，与 `kp_truth` 同级——
议程 `kp_text` 里会出现真相关键词（怪物、地点），提前泄露就毁了本模组。

## 2. `app/core/keeper/tools.py`

### 2.1 保留 key 与解析

```python
# keeper_state 里的系统保留 key：议程触发记账由代码写，不交给 LLM 的
# 自由记账（state_updates）——实测它多数轮根本不记，而 once 语义要靠它。
AGENDA_FIRED_KEY = "已触发议程"


def load_fired_agenda(keeper_state: dict | None) -> list[str]:
    """从状态笔记里解析已触发的议程 id（纯函数，无 IO）。

    存储形态是逗号分隔字符串——keeper_state 的值一律是 str（update_state_impl
    的契约），不为一个列表破例。
    """
```
要求：`None` / 缺 key / 空串 / `"a, b ,"` 都能正确解析成 `[]` / `["a","b"]`（去空白、去空项、保序）。

### 2.2 写入

```python
async def mark_agenda_fired_impl(deps: KeeperDeps, event_ids: list[str]) -> str:
    """把议程事件标记为已触发（幂等：已在列表里的忽略）。返回可读执行报告。"""
```
- 必须走 `deps.write_lock`（与 `update_state_impl` 一致——JSON 列是整体
  重新赋值，读改写并发会丢更新，这条在 v1 冒烟里真的踩过）；
- 读现值 → 合并去重保序 → 整体重新赋值 `room.keeper_state`（照抄
  `update_state_impl` 的写法，**不要**原地 mutate dict）；
- 用 `_record` 留痕，`event_type="keeper.agenda"`，payload 含 `event_ids`；
- 返回文本形如 `议程事件触发：{id}（{title}）`，多条用「、」连接。

🔴 `keeper.agenda` **不要**加进 `agent.py` 的 `_EVENT_LABELS` 和历史查询
白名单——议程发生了什么已经写在当轮叙事里，历史再重复一遍是冗余；
"哪些已触发"由第 4 节的状态块专门注入（与 `keeper.state` 同样的处理）。

## 3. `app/core/keeper/decision.py`

### 3.1 契约扩展

```python
class KeeperDecision(_DecisionModel):
    ...
    agenda_fired: list[str] = Field(
        default_factory=list, description="本轮真正发生的议程事件 id（不预告）"
    )
```

### 3.2 `execute_side_effects` 新增第三段（在 hp / state_updates 之后）

伪代码：

```
fired_now = []
already = load_fired_agenda(<当前 keeper_state>)
for eid in decision.agenda_fired:
    event = deps.module.agenda_by_id(eid)
    if event is None:
        issues.append(f"议程事件未执行：剧本里没有 id={eid}")
        continue
    if event.once and eid in already:
        logger.debug(...)          # 幂等跳过，不记 issue（不是错误）
        continue
    fired_now.append(eid)
if fired_now:
    report.append(await mark_agenda_fired_impl(deps, fired_now))
```

`already` 从哪来：`execute_side_effects` 目前不读房间状态。**在
`mark_agenda_fired_impl` 内部做幂等去重**（它本来就要读现值），
`execute_side_effects` 只负责 **id 合法性**校验 + 调用；`once=False` 的
事件允许重复触发，所以去重必须只对 `once=True` 生效 ——
把 `once` 的判断也下沉进 `mark_agenda_fired_impl`（它拿得到 `deps.module`）。

即：`execute_side_effects` 只做「id 不存在 → issue」，其余全在 impl 里，
返回文本要能区分「新触发」和「已触发过（跳过）」，两种都进 report 但措辞不同。

## 4. `app/core/keeper/prompts.py`

### 4.1 议程状态块

```python
def format_agenda_status(module: ScenarioModule, fired_ids: list[str]) -> str:
    """每轮注入的议程状态：哪些还没发生、哪些已经发生过。

    为什么代码算而不是让模型从剧本全文里自己推：`once` 语义和"别重复触发"
    是硬约束，靠模型记忆等于没有。
    """
```
- 未发生区：`- {id} · {title}（{触发条件人话}）：{kp_text 首句或全文}`；
- 已发生区：只列 `id · title`（内容已经在历史里了，不重复占 token）；
- `module.agenda == []` 时返回空串（调用方据此整块省略）。

### 4.2 `format_turn_input` 新增参数

```python
def format_turn_input(
    keeper_state, history_lines, roster, player_nickname, utterance,
    agenda_status: str = "",   # 新增，默认空 → 整块不渲染
) -> str:
```
块位置：**世界状态笔记之后、游戏历史之前**。默认值保证旧调用点/旧测试不动。

### 4.3 裁决 prompt（`build_adjudicator_instructions`）

- 渲染的剧本全文自动带上开场/议程（第 1.3 节已做），无需额外拼接；
- **新增规则 7（注意现有文件里有两条编号都是 6，顺手改成 6/7，新规则编号 8）**：

  ```
  8. **议程与游戏内时间**：世界不只随玩家行动而动，还有自己的时间表。
     ①每轮维护 keeper_state 的「游戏内时间」（如"第2天 夜晚"）——用
       state_updates 写，剧情推进到新的时段就更新；
     ②局面块的「议程状态」列出尚未发生的事件；当某条的触发条件在本轮达成
       （game_night 到达对应夜晚 / manual 时机成熟），把它的 id 写进
       agenda_fired，并在 narration_guidance 里指示叙事者把这件事呈现出来；
     ③agenda_fired 只写"本轮真的发生了"的——不预告、不提前铺垫；
     ④已发生区里的事件不要再触发（once），也不要在叙事里当新事件重讲；
     ⑤议程事件**不依赖玩家在场**：玩家没去监视，事件照样发生，玩家事后
       才看到痕迹（这正是时间压力的来源）。
  ```
- 输出格式的 JSON 示例加一行：`"agenda_fired": ["some-agenda-id"],`
  （示例里的 id 一律用中性占位，不要泄露具体模组内容）。

### 4.4 叙事 prompt（`build_narrator_instructions`）

在「守住秘密」那条里补一句：

```
剧本里的【议程时间轴】是幕后时间表——未发生的条目一个字都不能提前透露
（连暗示都不行）；只有本轮裁决指引明确说"这件事现在发生了"时才写它。
```

5. 再发一轮同类发言，确认**不会重复触发**（once 生效）。

每个反直觉决定都写了理由）、类型标注齐全、中文注释、行宽随 ruff 配置。
