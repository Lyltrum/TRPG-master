## 0. 目标与 B/A 分期

主持的结构化模组），用《科比特先生》端到端验证。

**B/A 分期（重要，决定本步边界）**：
- **本步是 B**：组装层的产出目标 = **现有的 `ScenarioModule` 形状**
  （meta / kp_truth / player_intro / opening / agenda / nodes / npcs / endings /
  kp_guidance）。**keeper 代码一行不改**——它今天就能吃这个形状。
- **A 是下一步**：等 B 在真跑中暴露出老 schema 的具体缺口，再把组装层 + keeper
  消费层迁到三层新结构（实体表/状态机/密级配对）。**本步不做 A。**

🔴 **与探针的关键区别**：前面几轮探针**刻意不给 schema**（为了发现形状）。
本步**刻意瞄准已知的 `ScenarioModule` 形状**——这是 B 的定义，不是倒退。
组装层的 prompt 里**可以**、也**应该**给出目标 schema。

**本步验证的真正意义**：keeper 是为《追书人》（线性调查）建的，而《科比特

它是**缺口发现的仪器**，冒烟中观察到的每个别扭都要记下来。

## 2. 组装层（新增，本步的核心 LLM 工作 = "预处理 agent" 本体）

把片段 + 关系组装成 `ScenarioModule` JSON。**分阶段做**（keeper v2 的教训：
窄合同的多次调用 > 一次调用干所有事）。离线批处理，慢无所谓。

### 目标 schema（照 `module_loader.py` 的形状，逐字对齐）

```
ScenarioModule:
  meta: {id, title, era?, tone?, designed_players?, multi_player_note?}
  kp_truth: {summary, key_facts[]}
  player_intro: str                      # 唯一直接念给玩家的
  opening?: {scene?, script, kp_notes?}
  agenda[]: {id, title?, trigger:str(自由文本), kp_text, effects[], once}
  nodes[]: {id, title, kp_text, checks[], branches[], sub_node?, leads_to[]}
    checks[]: {skill, difficulty?, on_success?, on_failure?, on_fumble?, prerequisite?}
    branches[]: {condition, outcome, then[]?}
  npcs[]: {id, name, role?, kp_notes?, stats?}
  endings[]: {id, title, condition?, trigger?, text}
  kp_guidance: {键: 自由文本}
```

### 阶段划分（建议，可在守住"分阶段"前提下微调）

**阶段 1 · 实体归组**：LLM 读全部片段 + 关系，把每个片段**分配**给一个归宿：
某个 node / 某个 npc / 某个 ending / 顶层背景(kp_truth) / 顶层指引(kp_guidance) /
议程(agenda)。产出 `片段id → 归宿` 的映射。
- **直接消费聚合/层级/包含关系**（03 底层八类里属于"实体表"的那四类）：
  被这些关系连在一起的片段应归入同一实体。
- 🔴 **无孤儿**：每个片段必须有归宿（逆投影判据①）。分不掉的片段单独列出，
  不是静默丢弃。

**阶段 2 · 实体成形**：对每个实体归宿，LLM 把该组片段合成目标对象——
node 的 kp_text/checks/leads_to、npc 的 kp_notes/stats、ending 的 condition/text。
- `leads_to`/`branches` 从**时序/因果/并列/路径关系**（属于"状态机"的四类）来；
- `checks` 从片段里描述"要检定"的内容抽出，**skill 用中文技能名**（下一步要
  对 COC7 规则表校验）。

**阶段 3 · 顶层字段**：kp_truth（从"守密人绝密"的背景片段）、player_intro
（**只能用玩家可见的公开信息**）、meta、kp_guidance、agenda。

**few-shot**：组装层可以在运行时读 `模组资料/追书人.structured.json`
（gitignored）当**格式范例**，让产出形状对齐。🔴 但《追书人》的内容**绝不能**
串进《科比特先生》的产物，也不能进任何 git 文件。

## 3. 校验闭环（新增，代码机械校验 + 自修重试）

产出 JSON 后跑校验；有错就把**错误清单**喂回组装层 LLM 重试（≤2 次）。
校验项（全部机械，不靠 LLM 自觉）：

| 校验 | 判据 | 依据 |
|---|---|---|
| **schema 合法** | `ScenarioModule.model_validate` 通过 | pydantic |
| **引用闭合** | `leads_to` / `sub_node.id` / `branches` 引用的 node id 都存在；ending/npc id 唯一 | 03 骨架层；pass1 真的引用过清单外 id |
| **技能名可查** | 每个 `check.skill` 能在 COC7 规则表（`build_coc7_ruleset()`）里解析到 | 检定发不出来 |
| **无孤儿片段** | 每个源片段 id 都出现在阶段1的归组映射里 | 逆投影判据①（03） |
| **不泄密** | 从 `kp_truth.key_facts` 抽出的真相关键词，在 **`player_intro` 与 `opening.script`** 里**零出现**（这两个是唯一直接给玩家的字段；node.kp_text 等本就是 KP 专用、含真相是正常的，不校验它们） | 分区层是 V，泄密不可逆 |

🔴 **不泄密校验的范围要精确**：只查 `player_intro` + `opening.script` 两个
玩家可见字段。把 kp_text 也当泄密来查是错的——它们本来就装真相。

自修循环：校验失败 → 把失败项文字化喂回 LLM → 重新产出 → 再校验。2 次仍不过
就**保存当前产物 + 完整校验报告**退出（不假装成功），让我看差在哪。

## 4. 验证：keeper 真能主持科比特先生（= 缺口发现）

产出 `模组资料/科比特先生.structured.json` 后：

1. **加载**：`load_module` 成功，打印 nodes/npcs/endings/agenda 计数；
2. **keeper 冒烟**（真实 DeepSeek，`.env` 已配 key）：把 `KEEPER_MODULE_PATH`
   指向新产出，建一间干净临时房 + 一名玩家 + 角色卡，发 2-3 轮玩家行动
   （至少一轮该触发检定、一轮纯对话），跑通 裁决→执行→叙事；
   - 🔴 **keeper 代码不许改**。冒烟目的是验证老 schema 装下科比特后 keeper
     能不能主持它；
3. 🔴 **缺口记录**（本步最有价值的产出，写进 `模组资料/科比特先生.B验证观察.md`，
   gitignored）：冒烟中每一处"老 schema 撑不住科比特"的地方——
   - 战斗/多敌人 HP 有没有地方放？塞进了哪、丢了什么？
   - 宅邸房间的邻接关系被 `leads_to` 有向链压成了什么样？
   - 同一实体的多片段（外观/行为/数据）聚合后 keeper 用起来顺不顺？
   - 玩家可见/绝密的配对在老 schema 里怎么表达的、够不够？
   这些就是 4b（A）的需求清单。**如实记，别粉饰"跑通了"。**
