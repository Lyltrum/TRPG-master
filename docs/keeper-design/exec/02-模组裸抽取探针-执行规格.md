# 执行规格 · 模组「裸抽取」探针（契约发现第 1 步）

> 设计依据：[03-模组抽象契约.md](../03-模组抽象契约.md) 第三节「契约是被发现的，
> 不是被设计的」与第六节路线第 1 步。
> 分支 `feat/keeper-agent`（实验分支，不开 PR、不碰 upstream）。

## 0. 这不是模组解析 Agent

**它的目的不是把模组解析好，是看模组自然吐出什么形状。**

现有 schema（`ScenarioModule`：nodes / npcs / checks / endings / leads_to）
完全是从《追书人》一个短篇线性调查模组倒推的。要证伪它，必须拿一个结构
不同的模组，**在不给它任何 schema 的情况下**看内容自己聚成什么类。

🔴 **因此本探针的第一原则：绝不预设类型名。** 一旦 prompt 里出现
「节点/NPC/检定点/结局/线索链」这些词，模型就会照着填，产出的是自我实现的
预言——用《追书人》的形状去看新模组，当然处处合身。

## 1. 输入与产物

| | 路径 | 进 git？ |
|---|---|---|
| 原始模组 | `模组资料/科比特先生.doc` | ❌ 目录整体 gitignore |
| 纯文本 | `模组资料/科比特先生.txt`（已用 `textutil -convert txt` 生成） | ❌ |
| 抽取产物 | `模组资料/科比特先生.裸抽取.json` | ❌ |
| 覆盖率报告 | `模组资料/科比特先生.覆盖率.md` | ❌ |
| **探针脚本** | `trpg-backend/scripts/module_probe/` | ✅ **方法要可审计**（同 `scripts/coc7_occupations/` 的先例） |

🔴 模组原文**绝不能**出现在任何进 git 的文件里——包括脚本里的示例、
docstring、commit message。脚本只吃路径参数，不内嵌任何模组内容。

## 2. 分块（必需，不是可选）

正文约 5.3 万字符（中文，估 4–7 万 token），单次调用装不下，长上下文里
抽取质量也会掉。

- 读 `.txt` 后**先给每行编号**（1-based），后续一切引用都用行号；
- **按结构边界分块，不按字数硬切**——切在句子中间会把条目切碎。
  先探查文本的实际结构（标题行、空行、编号段落的特征），据此定边界；
  **不要凭空假设格式，先打印几行看**；
- 每块 ≤ 6000 字符（超过就在最近的段落边界断开）；
- 每块附带**只读上下文**：前一块末尾约 200 字 + 后一块开头约 200 字，
  prompt 里明确标注「这段只用于理解上下文，**不要**为它产出条目」——
  跨块的内容不丢，又不会被抽两次。

## 3. 抽取 prompt（本规格的核心，逐字要求）

对每一块，用 DeepSeek（`deepseek-chat`，JSON mode，`temperature=0.2`——
抽取要稳定不要创造）问：

**允许出现的概念**：只有「这块内容在讲什么」和「它是给谁看的」。

**禁止出现的词**：节点、场景节点、NPC、检定、技能检定、线索、线索链、
结局、分支、leads_to、schema、字段——以及任何暗示目标结构的措辞。
（描述模组内容时不可避免会出现"人物""地点"这类普通词，指的是**不要把
它们当成要填的类型格子**。）

每个条目输出：

```jsonc
{
  "id": "模型自己起的短标识（英文小写连字符）",
  "title": "一句话标题",
  "what_kind_of_thing": "用你自己的话说，这是一段什么东西",
  "summary": "这条在讲什么，两三句",
  "audience": "谁能看到这段内容",
  "line_start": 120,
  "line_end": 147
}
```

三个字段的设计意图（**实现时必须保留，这是整个探针的价值所在**）：

- 🔴 **`what_kind_of_thing` 是自由文本，不是枚举。** 形状就是从这里浮现的：
  几十条自由描述去重之后聚成几类，那几类才是契约该有的骨架。给枚举等于
  提前把答案写进题目。
- 🔴 **`audience` 也是自由文本**，prompt 里可以说"通常是玩家可见或
  KP 绝密"，但**不强制二选一**。如果模型反复造出第三态（例如"KP 决定
  何时给玩家"），那本身就是发现——说明分区层需要三态而不是两态。
- **`line_start`/`line_end`** 是覆盖率的依据，必须是本块内的真实行号。

**不要**输出 type、category、node_id、check、difficulty 这类字段。
模型若自作主张多给，原样保留（`extra` 不丢），但 prompt 不得索取。

## 4. 覆盖率检查（防静默丢失，第一版就要有）

LLM 抽取最典型的翻车是**吐出看起来合理的 JSON、却漏掉半个章节**。
而"自然形状"的判断前提是抽取是完整的。

脚本机械计算并写进 `模组资料/科比特先生.覆盖率.md`：

- 总行数、被至少一个条目覆盖的行数、**覆盖率百分比**；
- **未覆盖的行号区间清单**，每段附该区间**开头 30 字**用于诊断
  （只在 gitignored 的报告里，不进 git、不贴进对话）；
- 重叠情况：被 2 条以上条目同时覆盖的行数（跨块重复抽取的信号）。

## 5. 汇总产出（我验收时最想看的）

除 JSON 外，在覆盖率报告里同时给出：

1. **条目总数**；
2. **`what_kind_of_thing` 的去重清单**——把所有自由描述列出来，相同的合并
   计数。🔴 **这是本轮最重要的产物**，形状从这里读出来；
3. **`audience` 的取值分布**（看是否自然二分，还是冒出了第三态）；
4. 每块的条目数（看有没有某一块明显抽少了 = 那块可能被草率处理）。

## 6. 明确不做（全部要等契约定了才做）

- ❌ 不给 schema、不预设类型
- ❌ 不做引用闭合校验（还没有契约可校）
- ❌ 不做两遍独立抽取 + 对拍（那是提质量的手段；第一版要的是自然形状，一遍够）
- ❌ 不做自修循环
- ❌ 不碰 `app/core/keeper/` 任何代码、不碰 SDK/前端/DTO/WS/alembic
- ❌ 不写测试（这是探针不是产品功能；脚本要能跑通即可）

## 7. 运行与验证

脚本用 `trpg-backend/.venv` 里已有的 `openai` SDK 和 `DEEPSEEK_API_KEY`
（`.env` 已配好，读法照抄 `app/core/config.py` 或直接 `os.environ`）。

跑完要报告（**每条输出真实可见，不要重定向到 /dev/null、不要串长 && 链**）：

1. 脚本实际执行的命令与耗时、调用次数、大致 token 消耗；
2. 覆盖率数字；
3. 🔴 **`what_kind_of_thing` 去重清单原样贴出来**（这是给我做契约判断的原料，
   不要总结、不要归类——归类是我的活，你归了就污染了）；
4. `audience` 分布；
5. 抽取过程中失败/重试的块。

`ruff check` 与 `ruff format --check` 要过（脚本在 `trpg-backend/` 下，
受同一套 lint 约束）。
