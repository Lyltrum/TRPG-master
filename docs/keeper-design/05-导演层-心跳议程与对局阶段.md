# 05 · 导演层：世界心跳、议程时钟与对局阶段（设计定稿，未实现）

> 用户真人实测后的根本性反馈："做出来的更像聊天机器人，不像 AI 主持人。"
> 诊断结论：**架构没错，但只建了一半**——两阶段回合制(04)解决的是"裁判"
> (裁决/掷骰/守秘/忠实度，已验证)，"导演"压根不存在。本文档是导演层的
> 完整设计，三个提案已获用户确认(2026-07-23)。

## 诊断：聊天机器人感的四个来源

| 真人主持人 | 现状 | 缺失机制 |
|---|---|---|
| 在时间里存在(玩家商量时插话、磨蹭时天黑) | 不发消息它就不存在 | **主动发言权** |
| 拥有流程(开场仪式→调查→高潮→结局→复盘) | 每步都要玩家拽着走 | **对局阶段状态机** |
| 导演场景("三天后，你们在档案室碰头") | 场景只随玩家要求切换 | **场景框架权** |
| 主持一桌人(点名、分聚光灯、反问) | 谁发消息回谁 | 群体主持(多人后补) |

**主持人 = 裁判 + 导演。** 裁判层(04)不动，导演层叠在它上面。

## 提案一：议程标注（模组 schema 长出时间维度）

JSON 不"差"，但没有时间结构——"时间压力"躺在 `kp_guidance` 自由文本里，
agent 读得到、执行不了。schema 新增(module_loader 宽松模型，向后兼容)：

```jsonc
{
  "opening": {                     // 开场脚本(阶段状态机的 opening 用)
    "scene": "委托人来访",
    "script": "念给玩家的引子(可基于 player_intro 扩写)",
    "kp_notes": "开场要建立什么(委托内容/初始线索/氛围基调)"
  },
  "agenda": [                      // 时间轴事件(世界心跳的燃料)
    {
      "id": "agenda-event-1",
      "trigger": { "type": "game_night", "at": 2 },   // 第2个游戏内夜晚
      "kp_text": "某个到点就发生、不依赖玩家行动的世界事件",
      "effects": ["世界状态发生改变", "若玩家在场监视→转入对应分支"],
      "once": true
    },
    {
      "id": "agenda-event-2",
      "trigger": { "type": "game_night", "at": 3 },
      "kp_text": "调查窗口关闭类事件(某条路线此后不可达)",
      "effects": ["此后某路线不可达，结局收束"],
      "once": true
    }
  ],
  "endings": [{ "trigger": "结构化条件(见提案三)", ... }]
}
```

trigger 类型第一版只做三种：`game_night`(游戏内第 N 晚，依赖 keeper_state
里的游戏内时间——裁决器已经在记)、`silence`(现实沉默 N 秒，心跳专用)、
`manual`(KP 裁量，agent 自行判断时机)。**具体模组的补标内容都来自其原文里
本就写着的时间压力**(某某夜发生某事、某窗口即将关闭)，只是把散在自由文本里的
东西结构化。

## 提案二：世界心跳（主动发言权）

**这是去聊天机器人化的关键机制。** 玩家不说话，世界也要动。

### 触发条件（全部满足才跳）
1. 房间 phase == InGame 且有活跃 WS 连接(没人在线不跳，省钱)；
2. 距最后一条消息(任意方向)沉默 ≥ `HEARTBEAT_SILENCE_SECONDS`(建议 90-120s)；
3. 无待掷检定(pending 队列空——等玩家掷骰不算冷场)；
4. 行动锁空闲；
5. 节流：距上一次**主动轮**≥ `HEARTBEAT_MIN_INTERVAL`(建议 5 分钟)，且
   连续主动轮 ≤ 2 次(玩家真离开了就安静等，不刷屏)。

### 技术形态
- 全局单个 ticker task(asyncio，应用 lifespan 启动)，每 30s 扫一遍活跃
  房间——不是每房间一个 task(房间数增长时任务失控)；
- 触发时走**同一套裁决→叙事管线**：合成一条 utterance="（时间悄然流逝）"
  的轮次，裁决器拿到额外指令"这是主动推进轮：按议程检查该发生的事件；
  没有议程事件时只推一小步(NPC 一句催促/环境一点变化/一声异响)，
  不许大幅推进剧情、不许替玩家行动"；
- 拿行动锁(拿不到=玩家恰好在提交，放弃本次心跳)；
- 产出照常广播 narration.push(前端零改动)；events 记 `keeper.heartbeat`
  留痕；
- 开关：`KEEPER_HEARTBEAT_ENABLED`(默认关，实验期显式开启)。

### 风险与克制
- **骚扰感**：主动轮的文案必须短(prompt 限 80 字内)、低频(节流参数)、
  可被玩家一句话打断；
- **成本**：每次心跳 = 2 次 LLM 调用，节流参数就是成本闸门；
- **幻觉推进**：主动轮裁决器 checks 必须为空(主动轮不发起检定——检定是
  对玩家行动的回应)，state_updates 只允许时间/氛围类。

## 提案三：对局阶段状态机 + 结局侦测

```
opening ──(开场仪式完成)──▶ investigation ──(结局条件满足)──▶ ending ──▶ 复盘
```

- **阶段存哪**：`keeper_state["对局阶段"]`(先复用状态笔记，不加列——
  实验期可逆决定)；
- **opening**：game.start 后第一轮走开场仪式——按模组 `opening.script`
  念引子、NPC 登场、递委托；仪式完成(裁决器判断"委托已建立")→ 进
  investigation；
- **结局侦测**：裁决器输出契约(KeeperDecision)新增可选字段
  `ending_reached: str | None`(结局 id)——每轮裁决顺带判断"是否满足某个
  结局的 trigger"；命中 → 叙事阶段收束(允许超长度上限的终章文案)→
  keeper_state 标记对局结束 → 后续行动回"本局已结束"；
- **endings[].trigger 结构化**：第一版不做完全形式化(条件判定本来就是
  LLM 裁量的强项)，只要求每个结局写清可判定的触发描述+裁决器显式输出，
  比"散文里藏条件"强一个数量级；
- 复盘接现有 `GET /rooms/{id}/summary` 桩(内容生成后续再议，不在本三案内)。

## 实施顺序与验收

1. **议程标注**(纯数据+loader)：《追书人》JSON 补 opening/agenda；
   loader 模型扩展+fixture 更新。验收：渲染进 system prompt，裁决器能
   引用议程事件。
2. **世界心跳**(核心)：ticker+触发条件+节流+主动轮管线。验收：冒烟里
   静默后守秘人主动说话且≤80字；连续沉默不刷屏；有待掷检定时不跳；
   e2e 零感知(默认关)。
3. **阶段状态机**：开场仪式+ending_reached 字段+收束流程。验收：新房间
   第一轮是开场仪式而不是干等玩家；冒烟里凑齐"对话"结局条件后守秘人
   主动收束。

依赖关系：提案二依赖提案一(议程是心跳的燃料)；提案三独立，可并行。
