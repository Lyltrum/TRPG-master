# 04 · 两阶段回合制架构（v2：裁决与叙事分离）

## v1 为什么被推翻

v1 = openai-agents 自由工具调用：一次 LLM 调用同时承担**理解/裁决/记账/叙事**
四个认知任务。实测四类 bug 同一病灶——模型的写作本能碾压其余三件事：

| 症状 | 曾经的补丁 | 结果 |
|---|---|---|
| 该掷不掷、线索白给 | 三轮 prompt 强化 → tool_choice=required + declare_no_check | 缓解但仍钻空子（"追问方式"拖延） |
| 掷了骰藏数字 | 代码强制附加 🎲 行 | 有效（保留进 v2） |
| NPC 现编、否认剧本 | 剧本全文常驻 prompt | 有效（保留进 v2） |
| 状态从来不记 | prompt 反复强调 | 无效 |
| Sonnet 模拟玩家 8 轮实测 | — | 整局仅 1 次掷骰（玩家显式喊技能那次） |

结论：**补丁都在对抗模型本能，而不是顺着设计**。真人 KP 是"先低头掷骰记账
（幕后）、再抬头讲故事（台前）"——两个分开的认知动作，v1 把它们压成了一个。

## v2 架构

```
action.submit
  ↓ 阶段1·裁决（LLM，JSON mode，temperature 0.3）
  │   输出 KeeperDecision：thinking / checks[] / san_checks[] /
  │   hp_changes[] / state_updates[] / narration_guidance
  │   ← 检定是 schema 字段而非"可选工具"，不存在"忘了裁决"
  │   ← thinking + 全部字段进 structlog，逐轮可审计
  ↓ 阶段2·执行（纯代码，decision.execute_decision）
  │   逐项调 tools.py 的 *_impl：服务端权威掷骰/写库/events 留痕
  │   非法项（未知技能等）跳过记为 issue，不炸整轮
  ↓ 阶段3·叙事（LLM，temperature 0.8，无工具无裁决压力）
  │   输入 = 局面 + 裁决指引 + 掷骰结果；写作本能从对抗对象变成生产力
  ↓ 代码强制附加 🎲 检定行（可见性硬保证，继承 v1）
```

对外不变：`Narrator` 接口 / WS 协议 / 锁 / 前端零感知。openai-agents SDK
退出主路径（依赖暂留）。

## 首轮实测对比（同样的隐式动作测试）

| 测试 | v1（8 轮实测） | v2（冒烟） |
|---|---|---|
| "我在书房仔细翻找"（不喊技能名） | 零检定，线索白给 | **裁决"命中剧本检定点"→ 侦查 1/70 大成功 → 给出剧本标注的日记** |
| "跟看守闲聊套近乎"（不喊技能名） | 零检定，纯叙事 | **裁决引用剧本节点 → 话术 44/60 成功** |
| 状态记账 | 全局 0 次 | 每轮都写 `当前场景` |
| 泄密/菜单/markdown/替玩家 | 偶发 | 冒烟零出现 |

## 关键设计点

1. **裁决温度 0.3、叙事温度 0.8**——裁决要一致性，叙事要创造力，一个调用
   一个温度在 v1 里根本做不到。
2. **裁决 JSON 可代码校验**（02 文档的落地）：执行器解析技能名对照 ruleset，
   非法项打回为 issue；未来两段式（01 文档）把"执行"换成发 `check.request`
   等玩家摇骰，插在阶段 2 的位置严丝合缝。
3. **防御模型偷懒**：`KeeperDecision.model_validate_json("{}")` 合法（= 本轮
   零操作），解析失败喂回错误重试一次，再失败才报 INTERNAL_ERROR。
4. **代价**：每轮 2 次 LLM 调用（原 1-2 次），延迟略增；DeepSeek 成本可忽略。
