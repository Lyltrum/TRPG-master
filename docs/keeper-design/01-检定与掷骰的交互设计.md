# 01 · 检定与掷骰的交互设计（A 问题）

## 问题

玩家想"亲手摇骰"，或线下真人围桌摇实体骰。当前实现把一次检定**塌缩**成
单段式：`action.submit` → agent 一次运行里调 `roll_check` 内部掷完 → 结果
直接写进叙事文本 → 广播。玩家没有参与掷骰的动作，只看到结果。

要展开成"主持人发起 → 玩家投 → 出结果 → 主持人据此续判"的两段式。

## 硬约束：不能在工具内阻塞

天真做法是让 `roll_check` 工具发出 `check.request` 后 `await` 玩家摇骰结果
再返回。**这条路被架构否决**：

- 当前模型是"每条 WS 消息一个短数据库事务、消息之间不持锁"（见
  `app/controller/ws.py` 顶部注释，SQLite 锁争抢的教训）。
- 房间行动锁（`action_lock.py`）在一次 `action.submit` 的 try/finally 里
  持有。工具内 `await` 真人操作 = 锁被真人手速持有，一个人发呆全房间卡住，
  时长完全不可控。
- 线下模式更极端：必然要等真人报数，工具内阻塞会把锁占到天荒地老。

## 结论：多轮闭环（一次检定 = 两个独立的短闭环）

```
闭环①（玩家行动 → 发起检定）
  action.submit
    → agent 判定需要检定，但【不掷】
    → 输出结构化"检定请求"(technique/difficulty/target-player)
    → 本轮结束、释放锁
    → WS 发 check.request 给对应玩家

  ……玩家在自己的时间里摇骰（锁已释放，房间不卡）……

闭环②（掷骰结果 → 续判）
  check.roll（玩家触发）/ check.result（结果落库）
    → 作为新事件，触发下一轮 agent
    → agent 看到骰子结果，继续叙事/裁决
```

好处：

1. 每个闭环各自短持锁，和现有架构严丝合缝；
2. 正好复用 `check.request`/`check.roll`/`check.result` 三个协议槽位
   （#77 铺好、未接真实逻辑）的**原始设计意图**；
3. "检定请求"变成显式、可审计、**可代码校验**的结构化事件——这反过来让
   B（发起权约束）可控（见 02）。

代价：agent loop 要改造成"这轮只发起、下轮再消费结果"的两段心智；prompt
和工具要为此重写。

## 掷骰执行：线上权威 vs 线下上报（房间级模式）

用户明确要两种都支持。它们在"服务端权威掷骰"这条全局原则上是**对立**的，
所以做成显式的房间级模式，而不是偷偷允许：

| 模式 | 谁掷 | 可作弊 | 复盘标记 | 适用 |
|---|---|---|---|---|
| `online-authoritative` | 后端 `dice.py` | 否 | `source: 'server'` | 线上局，防作弊 |
| `offline-reported` | 玩家物理摇后上报 | 是（靠信任） | `source: 'player-reported'` | 线下朋友围桌 |

关键：线下模式打破服务端权威不是 bug，是这个模式的性质（真人桌上也能偷改
骰子）。正确处理是**让它可见**——`check.result` 事件标注 `source`，复盘里
一眼看出"这颗骰子是玩家自己报的"。诚实 > 假装权威。

副产品：线下模式天生要等真人上报，**必然两段式**——又一次印证"多轮闭环"
是对的。

## 待观测（决定 A 的投入优先级）

A 是体验层，B 是地基。如果 agent 的检定发起判定（B）本就不可靠，那么再顺滑
的两段式也只是让玩家亲手摇一个"本不该掷的骰"。**先用单段式跑通、拿到 B 的
真实观感，再决定 A 怎么投入。**
