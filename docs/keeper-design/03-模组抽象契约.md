# 03 · 模组抽象契约

> 最终目的："所有模组都能抽象，让 AI 主持人推进游戏。" 这份文档把"抽象了
> 什么"讲成一个**契约**：模组必须提供什么、agent 保证什么、以及现在这个
> 契约有多少是猜的。

## 好消息：代码层的抽象已经在了

`keeper/` 里唯一跟《追书人》绑定的东西是 **JSON 数据文件**。`agent.py` /
`tools.py` / `prompts.py` 全部读 `deps.module`（`ScenarioModule` 实例）、
渲染任意模组，**零模组专属代码**。换模组 = 换一个 JSON，代码不动。

数据与代码的边界是干净的。这是抽象成立的必要条件，已满足。

## 契约

### 模组必须提供（`ScenarioModule` schema）

| 字段 | 作用 | agent 依赖它做什么 |
|---|---|---|
| `meta` | 标题/年代/基调/多人适配说明 | 定调、多人局归属规则 |
| `kp_truth` | 绝密真相 + 关键事实 | 守秘密的边界；它知道但不能说的东西 |
| `player_intro` | 唯一可公开的开场 | 开局念白 |
| `nodes[].checks` | **检定点：技能/难度/成败后果** | B 第一层护栏：照此发起检定，不自由发挥 |
| `nodes[].leads_to` | 线索链拓扑 | 节奏推进、判断玩家走到哪 |
| `npcs` | 名字/身份/动机/数据卡 | 扮演；专有名词以此为准（防现编） |
| `endings` | 结局条件 | 判定收束 |
| `kp_guidance.core_clue_paths` | 通往真相的冗余路径 | 线索不卡死（失败换代价，不断链） |

### agent 保证（无论哪个模组）

1. **守秘**：`kp_truth` 绝不原样吐出；线索靠玩家挣得；真相关键词挣得前零出现；
2. **照检定点发起**：命中模组 `checks` 时用给定技能/难度，不自由发挥；
3. **线索不卡死**：单次失败不断核心链，代价化处理（时间/资源/风险）；
4. **状态记账**：进展写 `update_state`，伤害/理智走 `adjust_hp`/`san_check`；
5. **不泄结构**：不出现"模组/剧本/工具"等幕后词汇。

## 坏消息：N=1，通用性是猜的

《追书人》是**线性调查模组**。当前 schema 的形状假设 = "节点 + 线索链 +
检定点"。以下模组类型能不能装进这个 schema，**完全没验证**：

| 模组类型 | 现 schema 的缺口（推测） |
|---|---|
| 战斗密集型 | 没有回合/先攻/多敌人 HP 追踪/战技的结构位；`npcs[].stats` 是自由字典，撑不起结构化战斗 |
| 非线性沙盒 | `leads_to` 是有向链，沙盒是"哪都能去"的图，节奏推进逻辑不同 |
| 资源/时间管理型 | 没有"倒计时/资源池/触发器"的一等结构（追书人的时间压力是塞在 `kp_guidance` 自由文本里的，agent 未必接得住） |
| 理智/神话密集型 | San 触发点散在 `checks.on_success` 文本里，没有独立的"目击→San 损失表"结构 |
| 多线并行 | 没有"支线状态机"的表达 |

**结论：真正的抽象要从第二个结构不同的模组里长出来。** 现在宣称"所有模组
可抽象达成"是自欺。正确姿势：

1. 先用《追书人》把单段式玩通（当前）；
2. 找一个**结构不同**的模组（首选战斗密集或理智密集，最能压垮现 schema）
   做第二次结构化——过程中 schema 会被迫长出新字段；
3. 两三个模组之后，共性稳定下来，才谈得上"通用契约"。

## 抽象的推进原则

- **数据驱动，代码不认模组**：任何模组专属逻辑一旦想写进 `keeper/*.py`，
  就是抽象失败的信号——应该改成 schema 字段 + 通用处理。
- **schema 宽松、渐进**：`module_loader.py` 用 `extra="ignore"`，允许模组带
  额外字段，加新结构不破坏老模组。
- **结构化环节承担 B 第一层**：把 agent 运行时最易错的"检定发起"判断，前移
  到"模组导入/结构化"这一步定死——这也是未来"模组导入 Agent"（架构蓝图里
  的 `moduleimport`）真正要自动化的活。
