# 03 · 模组抽象契约（v2：扩到整条链）

> 最终目的："所有模组都能抽象，让 AI 主持人推进游戏。"
>
> **v2 的范围变化（2026-07-24）**：这条链上不再只有主持人。用户决定在本分支
> 单干整条链——**模组原文 → 预处理 agent → 契约 → 主持人 agent → 导演层**。
> 契约因此从"主持人读什么"变成"两个 agent 之间的接口"，是整条链的窄腰。
>
> 本文写三件事：契约长什么样、契约怎么**被发现**（而不是被设计）、
> 预处理 agent 的职责边界与验收。

## 一、好消息：代码层的抽象已经在了

`keeper/` 里唯一跟《追书人》绑定的东西是 **JSON 数据文件**。`agent.py` /
`tools.py` / `prompts.py` 全部读 `deps.module`（`ScenarioModule` 实例）、
渲染任意模组，**零模组专属代码**。换模组 = 换一个 JSON，代码不动。

数据与代码的边界是干净的。这是抽象成立的必要条件，已满足。

## 二、契约的三层：分界线是「谁消费」

不是"结构化 vs 自由文本"，是**代码要拿它干什么**。

### ① 骨架层（代码消费，强约束）

凡是代码要拿来**记账 / 去重 / 查表 / 防幻觉**的，必须可寻址：

| 要求 | 为什么 |
|---|---|
| 每个可寻址实体（节点/NPC/议程条目/结局/检定点）有唯一 id | 记账与去重的把手（`agenda_fired` 的 once 就靠它） |
| 引用必须闭合（`leads_to`、结局 id、槽候选、技能名） | 悬空引用 = 运行时静默失效；**扫描手法项目里已有先例**（#114 槽候选集守卫） |
| 检定点的技能名在 COC7 规则表里查得到 | 规则引擎是后端权威，对不上就发不出检定 |
| NPC 专有名词表 | 防现编——v1 冒烟里 DeepSeek 自己发明过 NPC 名字 |

**代码对这一层的全部操作是：发号、记账、去重、留痕、查表。不理解语义。**

### ② 内容层（LLM 消费，一律自由文本）

触发条件、后果、动机、氛围、KP 指引、时间压力——**不做枚举、不做类型、
不做归一化**。

理由有两条，第二条是 v2 才想清楚的：

1. 主持人手里有剧本全文 + 完整历史 + 世界状态，判断"到没到时候"本来就该它做。
   代码去算反而脆——实测把游戏内时间写成「第2天 傍晚」而非「第2个游戏内
   夜晚」，模型就误触发了第一夜的事件。**那不是措辞问题，是代码本就不该
   去算第几晚。**
2. 🔴 **强类型字段会把语义归类的难题转嫁给预处理 agent。** `trigger:
   {type: "game_night", at: 2}` 这种形状，逼预处理去判断"第二个满月之夜"
   该归到哪个枚举——归错就废了，而枚举本身是从一个模组倒推的。
   **契约每强一分，预处理的失败面就大一分。**

### ③ 分区层（安全约束，唯一必须严格的语义要求）

每块内容标注 **玩家可见 / KP 绝密**。这是唯一不能松的一层——分错了主持人
再谨慎也会泄密（它读到的就是错的）。整个 keeper 实验里反复验证：秘密管理
是核心，泄密不可逆。

### 契约厚度的定则

> **由「代码机械操作的必需品」定下界，不由「主持人想要什么」定上界。**

契约越薄，两个 agent 越容易对齐。"预处理必须对齐主持人"是单向的说法；
实际上主持人也要克制，只依赖非要不可的东西。

## 三、契约是被发现的，不是被设计的

### 提案①的教训（如实留痕）

2026-07-24 做提案①（议程标注）时，我把 `trigger.type` 钉成了
`game_night` / `silence` / `manual` 三种——**全部从《追书人》一个模组倒推**。
`game_night + at: N` 之所以长这样，纯粹因为这个模组的时间压力恰好按"夜"计。
换成按小时推进的邮轮模组、按失败次数推进的模组、无时间压力的沙盒，都套不进去。

而本文档 v1 早就写着"真正的抽象要从第二个结构不同的模组里长出来"——
**做的时候没回头看自己写的原则。** 这就是"契约被设计出来"的典型症状：
它长得像最近看过的那个模组。

另有一处分类错误：`silence`（现实沉默 N 秒）根本不是模组的时间轴，是**引擎侧**
触发条件（世界心跳用），塞进模组 schema 属于把两类东西混进一个字段。

### 正确的发现方法

1. **第二个模组不要人工结构化。** 人工做的时候必然不自觉迁就已有 schema，
   得到的是自我实现的预言。
2. **让预处理 agent 裸跑第二个模组**——不给它现有 schema，只要求"带 id 的
   切分 + 分区"。看它**自然吐出什么结构**。
3. 烂的输出恰恰最有价值：它填不出来的字段，就是不该存在的字段。
4. 两个模组的自然形状 + 《追书人》的人工版三者取共性 = 骨架层；
   差异处 = 该留在内容层的自由文本。

## 四、预处理 agent：职责边界与验收

### 形态：流水线，不是 tool-calling loop

它跟 keeper 是两种东西：keeper 是回合制交互、要快；预处理是**离线批处理、
可以慢**，因此可以多轮自检、可以人工介入、可以跑两遍对拍。

```
原文 → 分块 → 逐块抽取 → 合并 → 校验 → 自修 → （人工抽检）→ 结构化 JSON
```

### 职责收窄为四件事

**切分 + 命名 + 分区 + 引用闭合。不做语义归类。**

### 复用已验证的手法：两遍独立抽取 + 对拍

#114 扩 229 个职业时用过并验证有效：两遍独立推导 → 对拍差异 → 口径类分歧
由代码机械消解 → 只有真分歧才需要人裁定（那次 64 条分歧里 60 条机械消解，
真正人工裁定 4 条）。模组抽取同构,可直接搬。

### 可机械验收的清单

| 验收项 | 手法 | 防的失败 |
|---|---|---|
| 引用闭合 | 悬空引用扫描 | 运行时静默失效 |
| id 唯一 | 平凡 | 记账串味 |
| 技能名合法 | 对 COC7 规则表 | 检定发不出来 |
| **内容覆盖率** | 原文每段是否被至少一个条目吃掉 | 🔴 **静默丢失**——LLM 总能吐出看起来合理的 JSON，却漏掉半个章节，这是抽取最容易翻车的地方 |
| 秘密分区 | 真相关键词在玩家可见区零出现 | 泄密 |

### 一条必须接受的现实

**真实模组的写作质量差异极大**，有的原文根本没有清晰的检定点或结局条件。
预处理不能无中生有。因此**契约的必填项要极少**，主持人要能在信息不全时工作
（它已经能——`kp_guidance` 就是自由文本兜底）。

## 五、当前 schema 的待证伪清单

《追书人》是**线性调查模组**，现 schema 的形状假设 = "节点 + 线索链 + 检定点"。
以下全部**未验证**，等第二个模组来打：

| 模组类型 | 现 schema 的缺口（推测） |
|---|---|
| 战斗密集型 | 没有回合/先攻/多敌人 HP 追踪/战技的结构位；`npcs[].stats` 是自由字典 |
| 非线性沙盒 | `leads_to` 是有向链，沙盒是"哪都能去"的图 |
| 资源/时间管理型 | 提案①加了 `agenda`，但 trigger 形状是从《追书人》倒推的（见第三节） |
| 理智/神话密集型 | San 触发点散在 `checks.on_success` 文本里，没有独立的"目击→San 损失表" |
| 多线并行 | 没有"支线状态机"的表达 |

## 六、路线（v2）

```
0. 拿到第二个结构不同的模组原文        ← 唯一的外部依赖
1. 预处理 agent 裸跑它（不给现有 schema）
2. 三份材料取共性 → 契约定稿（本文档第二节收敛）
3. 收缩 keeper schema 到契约：trigger 退自由文本、silence 挪进引擎配置
4. 预处理 agent 加契约约束 + 校验闭环（生成→校验→自修）
5. 回到导演层（提案②世界心跳 / ③阶段状态机）
```

**为什么导演层排最后**：心跳吃议程当燃料，议程形状没定就做心跳是白做。

**不依赖第 0 步、现在就能做的**：第 3 步里 trigger 的收缩（方向已确定，
且它同时修掉"代码去算第几晚"那个脆点）。

## 七、推进原则

- **数据驱动，代码不认模组**：任何模组专属逻辑一旦想写进 `keeper/*.py`，
  就是抽象失败的信号——应该改成 schema 字段 + 通用处理。
- **schema 宽松、渐进**：`extra="ignore"`，加新结构不破坏老模组。
- **强类型要付费**：每加一个枚举字段，就问一句"预处理 agent 填得出来吗？
  填错的代价是什么？"填不出来或填错很贵的，退回自由文本。
- **结构化环节承担 B 第一层**：把 agent 运行时最易错的"检定发起"判断前移到
  预处理阶段定死——这正是预处理 agent 真正要自动化的活（见 02）。
- **契约的每一条都要能说出"哪个真实模组逼出了它"**。说不出来的，就是猜的。
