# 02 · 检定发起权的分层约束（B 问题）

## 问题

"要不要检定、用哪个技能、什么难度"——这一步把不确定性交给 LLM，是最容易
翻车的地方（冒烟里已暴露"该掷不掷""乱编技能名"）。既要支持玩家主动宣告
技能，又不能让玩家专挑对自己有利的技能狂摇。发起权本质仍在 KP。

## 三层护栏（从强到弱）

### 第一层：模组标注的检定点（最强，模组进来时就定死）

`ScenarioModule` 每个节点的 `checks[]` 已带 `skill` / `difficulty` /
`on_success` / `on_failure` / `on_fumble` / `prerequisite`（见
`追书人.structured.json`）。agent 命中这些场景时**照模组来，不自由发挥**：
技能、难度、成败后果全部由数据给定，agent 只负责叙事包装。

这是发起可靠性的地基，也是"所有模组可抽象"时**每个模组都必须标注**的东西
（见 03 契约）。模组作者/结构化环节把检定点定好 = 把 agent 最易错的判断
从运行时挪到了导入时。

### 第二层：模组没覆盖的即兴动作（考察 agent 真本事）

玩家总会做剧本没写的事。这里只能靠 agent 临场判定。规则能约束的是**结果侧**
而非**发起侧**：

- 难度分级（普通/困难/极难）、成功等级换算 → 走确定性代码（`dice.py` 的
  `evaluate_check` 已实现）；
- agent 只决定两件事：掷不掷 + 哪个技能。这两件事无法用规则兜死，是 LLM
  判断力的真实考场——本轮实验重点观测的就是它。

### 第三层：玩家主动宣告技能时的合理性校验

玩家喊"我要用侦查" ≠ agent 照单全收。发起权仍在 KP：

- agent 要判断"这个技能在当前情境下合理吗"（在黑暗密室里宣告"图书馆使用"
  显然不合理）；
- 不合理 → 拒绝或改判成更合适的技能；
- 这条目前只能靠 prompt 约束（system prompt 已有"检定归行动宣告者/剧本没写
  的按 COC7 常识裁量"，需补强"玩家宣告的技能要过合理性关"）。

## 两段式（见 01）如何让 B 更可控

单段式下 agent 掷完直接写进叙事，你**连拦截的机会都没有**。两段式把检定请求
变成结构化事件 → 在 `check.request` 真正发给玩家**之前**，可以用代码校验：

- 这个 `skill` 在当前节点的模组 `checks` 里合法吗？
- 这个 `difficulty` 是模组给定的，还是 agent 擅自加码的？
- 非法 → 打回让 agent 重新判定，而不是让错误的检定发到玩家面前。

即：**A 的多轮闭环，天然是 B 的执行护栏。** 这是把两个问题分开看之后才浮现
的联系。

## 落地顺序

1. 先把三层护栏写进 prompt（第三层）+ 确认第一层数据驱动路径（模组 checks
   优先于即兴）；
2. 本轮单段式实验观测：命中模组检定点时照不照做、即兴判定准不准、玩家宣告
   不合理技能时挡不挡得住；
3. 拿到数据后，两段式落地时把"检定请求代码校验"作为第一层/第三层的运行时
   兜底。
