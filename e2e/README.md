# trpg-e2e

**SDK → 后端**的端到端验证：用真实 `trpg-sdk` 打真实后端，验证契约与业务流程。

**不含浏览器，也不依赖前端。**

## 为什么是这一层

三个单包 CI（backend / sdk / frontend）各自只覆盖各自那一层，集成缝隙没人守。曾经做过一版**浏览器** e2e，当天就撤了——前端正在因「数据源归位」实质重构，选择器会跟着 UI 反复碎。

而 **SDK 是当前和未来前端都要依赖的契约层**：前端重写、甚至换一个客户端，这些用例依然成立。它还便宜地做到了浏览器层做不划算的事：**在一个进程里起两个客户端**，所以「第二个人加入」「广播有没有到达另一个人」这类断言才做得起。

## 怎么跑

```bash
npm ci
npm run test:e2e
```

**不需要先手动起后端**——`scripts/run-e2e.ts` 会起后端、等就绪、跑测试、收摊，本地和 CI 走同一条路径（两套流程迟早分叉，然后出现「CI 绿本地红」）。

前置条件只有一个：`trpg-backend` 已经 `uv sync`（脚本直接调 `.venv/bin/` 里的可执行文件），以及 `trpg-sdk` 已经 `npm run build`。

后端跑在 **8099**（不是开发用的 8000），且**端口被占用会直接报错退出**，不会退化成「连到别人的后端上」。需要换端口用 `E2E_PORT=xxxx`。

只跑某个文件：

```bash
E2E_ONLY=tests/character-build.e2e.ts npm run test:e2e
```

## 覆盖了什么

| 文件 | 内容 |
|---|---|
| `character-build.e2e.ts` | ruleset 契约（9 属性 / `pointBuy` 标志 / 点数购买约束 / 年龄区间）、建卡全流程（草稿→保存→**读回**→完成）、**属性点预算按生成方法分流**、**掷骰来源被伪造时退回点数购买法**、年龄区间、信用必填、角色卡隐私 |
| `multiplayer-ws.e2e.ts` | 第二个玩家加入、WS `join → session.bound`、`game.start`、**行动广播到另一个客户端** |
| `not-implemented.e2e.ts` | 钉住目前还是桩的接口（复盘摘要 / 常用卡库），并确认 replay 是真实现 |

## 写用例的约定

- **断言具体规则，不是「接口能通」。** 一个只检查 200 的用例几乎抓不到真问题。
- **禁止固定 `sleep`。** 等事件用带超时的 `waitForEvent`，等后端就绪用轮询。
- **WS 一定要在 `finally` 里 `disconnect()`。** 用例失败时漏掉的话，WS 句柄会让 node 进程不退出，表现成「测试跑完了但命令挂住」——踩过。
- **每次跑用全新 `e2e.db`**，且不碰开发用的 `app.db`。
- **🔴 加了用例要做一次变异检验**：故意把它该守住的东西改坏，确认它**红得起来**。

  它还会顺带体检整套装置：本目录第一次做变异检验时，后端明明改坏了、13 条却全绿——原因是当时用 8000 端口，本地开着开发后端，脚本 spawn 的 uvicorn 绑不上端口秒退，而就绪探测只看「这个端口能不能应答」，于是**整套用例静默跑在了开发后端 + 开发库 `app.db` 上**。已改用 8099 + 端口占用直接报错 + 检测子进程退出。**HTTP 能应答，不代表应答的是你刚起的那个进程。**

  这不是形式主义——本目录的「掷骰法不受预算约束」那条，第一版直接拿掷出来的属性做断言，变异检验时**测试照样绿**：掷骰 8 项总和均值约 457，随机结果经常不到 480，根本没踩到「超预算」这个条件。

  ⚠️ 那一版的补救是"显式写死 8 项全 90"，**现在已经不能那么写了**：那正是 PR #97 堵掉的伪造手法（掷完骰再把属性整组换掉 → 来源退回 `pointbuy`）。现在的写法是**保留服务端的 `roll` 来源、最多重掷 50 次直到真实总和超过 480**；写死 90 的是它的对照用例（伪造来源后必须被预算拒）。这件事本身也值得记：**产品变更会作废测试的构造方式，而不作废它的意图**——照着过时的描述去"修"测试，很容易把防线改回去。

## 已知缺口（用例里遇到、但不属于本目录该修的）

- **`rooms.create` 不接受 token**：SDK 没给这个端点留鉴权入口，所以通过 SDK（以及走 SDK 的前端）创建的房间，`Room.host_user_id` 永远是 `null`。
- **`list_my_rooms` 按 `reconnect_token` 查、只返回一个房间**：「我的游戏」实际是「这个浏览器的最后一个房间」，不是「这个账号的所有房间」——而账号体系当初就是为了「换设备找回游戏」才引入的。
- **`join_room` 无幂等**：重复加入会给同一个人建重复 player 行；且 `Building` 阶段直接 409，正常重连进不去。
